1 — Rappels : dictionnaires et récursivité

Les dictionnaires Python

Un dictionnaire associe des clés à des valeurs. En Python, les clés sont généralement des chaînes de caractères, et les valeurs peuvent être n'importe quel objet — y compris… un autre dictionnaire. C'est précisément ce que tu vas manipuler ici.

Rappel — Opérations de base sur un dictionnaire 
# Accès à une valeur
d = {"Logement": 2660, "Transport": 708}
print(d["Logement"])      # 2660

# Parcours clé/valeur
for cle, val in d.items():
    print(cle, "→", val)

# Tester si une valeur est un dictionnaire
isinstance(val, dict)     # True ou False

Rappel sur la récursivité

Une fonction est récursive quand elle s'appelle elle-même. Pour éviter une boucle infinie, toute fonction récursive doit comporter :

Structure d'une fonction récursive
  • Un cas de base — la situation la plus simple, résolue sans appel récursif. C'est la condition d'arrêt.
  • Un appel récursif — la fonction se rappelle sur un problème strictement plus petit, garantissant qu'on atteindra le cas de base.
Exemple — Somme récursive d'une liste 
def somme(liste):
    if len(liste) == 0:     # cas de base : liste vide → somme = 0
        return 0
    return liste[0] + somme(liste[1:])   # appel récursif sur le reste

Pour somme([3, 1, 4]), Python calcule : 3 + somme([1, 4]) → 3 + 1 + somme([4]) → 3 + 1 + 4 + somme([]) → 3 + 1 + 4 + 0 = 8.

Récursivité sur des dictionnaires imbriqués

Quand les valeurs d'un dictionnaire peuvent elles-mêmes être des dictionnaires, la récursivité est l'outil naturel : si la valeur est un dictionnaire, on rappelle la fonction sur ce sous-dictionnaire ; sinon, on traite la valeur directement.

Schéma général 
def traiter(dico):
    resultat = 0
    for cle, valeur in dico.items():
        if isinstance(valeur, dict):     # valeur = sous-dictionnaire
            resultat += traiter(valeur)  # appel récursif
        else:                            # valeur = nombre
            resultat += valeur           # cas de base
    return resultat

À chaque appel récursif, on « descend » d'un niveau dans l'arborescence. Le cas de base est atteint quand la valeur n'est plus un dictionnaire — c'est une feuille de l'arbre.

Question 1 — Somme des valeurs agrégées

Énoncé — Question 1

Q1 Compléter le corps de la fonction total_simple(empreinte) qui prend en paramètre un dictionnaire à un seul niveau associant des catégories à des entiers, et renvoie la somme de toutes les valeurs.

Ajouter ensuite un appel permettant d'afficher l'empreinte totale d'Ada en chargeant le fichier agrégé.

Indice 1

Pour n'obtenir que les valeurs d'un dictionnaire (sans les clés), utilise dico.values(). Tu peux ensuite appeler directement sum() dessus, ou bien faire une boucle for.

Indice 2

Python permet d'écrire simplement : return sum(empreinte.values()). Vérifie sur le dictionnaire agrégé d'Ada : la somme doit valoir 7 252 (2660 + 1500 + 708 + 893 + 1491).

Éditeur Python — Complète la fonction
Correction — Question 1

Explication : dico.values() renvoie une vue sur toutes les valeurs du dictionnaire. sum() les additionne en une seule expression. Pour la forme agrégée, les valeurs sont toutes des entiers — pas de sous-dictionnaire — donc pas besoin de récursivité ici.

Vérification : \(2660 + 1500 + 708 + 893 + 1491 = 7252\) kg de CO₂ par an.

Question 2 — Somme récursive des dictionnaires imbriqués

Énoncé — Question 2

Q2 Écrire le corps de la fonction récursive total_rec(empreinte) qui calcule la somme de toutes les valeurs numériques présentes dans un dictionnaire dont les valeurs peuvent elles-mêmes être des dictionnaires (imbrication quelconque).

Utiliser la fonction est_dictionnaire(objet) fournie pour tester la nature de chaque valeur. Compléter les tests avec un cas proche de la structure réelle d'Ada.

Indice 1

Pour chaque paire (cle, valeur) du dictionnaire, il y a deux situations :

  • valeur est un dictionnaire → appel récursif sur ce sous-dictionnaire.
  • valeur est un nombre → c'est un cas de base, on l'ajoute directement.
Indice 2 — Trace d'exécution

Pour {"a": {"b": 1, "c": 2}, "d": {"e": 3}} :

total_rec({"a": {"b":1,"c":2}, "d": {"e":3}})
  → "a" est un dict → total_rec({"b":1,"c":2})
      → "b" = 1 (nombre) → +1
      → "c" = 2 (nombre) → +2
      → retourne 3
  → "d" est un dict → total_rec({"e":3})
      → "e" = 3 (nombre) → +3
      → retourne 3
  → retourne 3 + 3 = 6 ✓
Éditeur Python — Complète la fonction récursive
Correction — Question 2

Structure de la solution : la variable total accumule les sommes. Pour chaque paire clé/valeur, on teste si la valeur est un dictionnaire (est_dictionnaire) — si oui, appel récursif ; si non, addition directe.

Vérification croisée : total_rec sur le dictionnaire détaillé doit donner le même résultat que total_simple sur le dictionnaire agrégé : 7 252 kg. Si les deux ne concordent pas, c'est qu'une valeur est mal placée dans le JSON.

Cas de base implicite : si le dictionnaire est vide ({}), la boucle ne s'exécute pas et la fonction renvoie 0. Aucun test supplémentaire n'est nécessaire.

Question 3 — Analyser et corriger le bug

Énoncé — Question 3

Q3 La fonction alerte_valeur_aberrante(empreinte, limite) ci-dessous est censée renvoyer True dès qu'une valeur numérique du dictionnaire (à n'importe quelle profondeur) dépasse strictement la limite.

Exécutée sur le dictionnaire complet d'Ada avec limite = 1000, elle renvoie False — alors que le chauffage individuel s'élève à 1 500 kg.

Expliquer précisément l'origine de l'erreur, puis proposer une version corrigée.

Voici le code fourni :

def alerte_valeur_aberrante(empreinte, limite):
    for categorie, valeur in empreinte.items():
        if est_dictionnaire(valeur):
            return alerte_valeur_aberrante(valeur, limite)  # ← BUG
        else:
            if valeur > limite:
                return True
    return False
Indice 1

Quand Python rencontre return dans une boucle for, il quitte immédiatement la fonction — la boucle s'arrête, même si des éléments n'ont pas encore été examinés.

Ici, dès que la première valeur du dictionnaire de niveau 0 est un sous-dictionnaire, la fonction renvoie directement le résultat de l'appel récursif sur ce seul sous-dictionnaire — sans jamais examiner les catégories suivantes.

Indice 2 — Trace

Le dictionnaire d'Ada commence par "Logement", dont la valeur est un sous-dictionnaire. Dès la première itération, la version buguée fait : 

return alerte_valeur_aberrante(empreinte["Logement"], 1000)

L'exécution plonge dans "Logement". La première clé de ce sous-dictionnaire est "Energie" (encore un dict), donc : 

return alerte_valeur_aberrante(empreinte["Logement"]["Energie"], 1000)

Là, "Electricite" = 206 ≤ 1000 (pas d'alerte), "Cuisson" = 105 ≤ 1000 (pas d'alerte), "Chauffage individuel" = 1500 > 1000 → renvoie True.

Attends — dans ce cas précis, ça trouve bien le chauffage ! Le vrai problème apparaît si la première catégorie ne contient aucune valeur aberrante : la fonction renvoie False sans jamais regarder les autres catégories.

Essaie avec limite = 400 : seule la viande rouge (814), le chauffage (1500), et quelques autres dépassent. La fonction buguée peut renvoyer False si les valeurs aberrantes se trouvent dans les catégories examinées après le premier retour.

Éditeur Python — Observe le comportement bugué
Correction — Question 3

Origine du bug : la ligne return alerte_valeur_aberrante(valeur, limite) provoque un retour immédiat dès la première sous-catégorie rencontrée. Si ce sous-dictionnaire ne contient pas de valeur aberrante, la fonction renvoie False sans avoir examiné les catégories suivantes.

Il faut distinguer deux cas pour le résultat récursif : si l'appel récursif renvoie True, on peut rentrer immédiatement ; si il renvoie False, on doit continuer la boucle pour examiner les catégories restantes.

La correction en un mot : remplacer return appel_recursif(...) par if appel_recursif(...): return True.

La subtilité : on ne peut rentrer immédiatement que si on a trouvé une alerte (True). L'absence d'alerte dans un sous-dictionnaire (False) ne signifie pas que tout le dictionnaire est correct — il faut continuer la boucle pour vérifier les autres branches.

Question 4 — Concevoir un jeu de tests robuste

Énoncé — Question 4

Q4 Proposer un jeu de tests pertinent pour la fonction alerte_valeur_aberrante corrigée. Pour chaque cas de test, préciser :

  • la structure du dictionnaire fourni en entrée,
  • le résultat attendu,
  • la particularité algorithmique que ce test vérifie.
Méthode — Comment concevoir de bons tests ?

Un bon jeu de tests doit couvrir les cas limites (valeur exactement égale à la limite, dictionnaire vide, un seul élément) et les cas structuraux (valeur aberrante en surface, en profondeur, dans la dernière branche). Il faut aussi un cas où la réponse est False pour s'assurer que la fonction ne renvoie pas toujours True.

Réfléchis à ce qui pourrait casser la fonction : une valeur exactement égale à la limite, une valeur aberrante dans la dernière catégorie (pas la première), une imbrication profonde…

Pièges fréquents & astuces de vérification

Piège 1 — return dans une boucle coupe tout

C'est le cœur du bug de la Question 3. Dans une boucle for, return valeur quitte immédiatement la fonction — même si des éléments n'ont pas encore été visités. Pour un appel récursif qui peut valoir False, il ne faut jamais écrire return appel_recursif(...) directement dans la boucle. Il faut d'abord tester le résultat : if appel_recursif(...): return True.

Piège 2 — Confondre « supérieur » et « supérieur ou égal »

L'énoncé dit « valeur strictement supérieure à la limite ». La condition correcte est valeur > limite, pas valeur >= limite. Une valeur exactement égale à la limite ne doit pas déclencher l'alerte. C'est pour ça que le Cas 4 du jeu de tests est indispensable.

Piège 3 — Oublier le cas de base de la récursivité

Dans total_rec, le cas de base n'est pas explicitement écrit sous forme de if len(dico) == 0: return 0. Il est implicite : si le dictionnaire est vide, la boucle for ne s'exécute pas et la fonction renvoie 0 (valeur initiale de total). Il faut comprendre pourquoi ça fonctionne.

Piège 4 — Utiliser sum() sur des dictionnaires imbriqués

sum(empreinte.values()) fonctionne pour total_simple car les valeurs sont toutes des entiers. Pour total_rec, les valeurs peuvent être des dictionnaires : sum() lèverait une TypeError en tentant d'additionner un entier et un dictionnaire. La récursivité est obligatoire dès qu'il y a des niveaux imbriqués.

Mini-exercice A — Calcul mental récursif

Sans Python, calcule total_rec(d) pour :

d = {"X": {"a": 10, "b": {"c": 5, "d": 3}}, "Y": 20}

d["X"] est un dict → on descend :
  "a" = 10 (feuille)
  "b" est un dict → on descend :
    "c" = 5, "d" = 3 → 8
  sous-total X = 10 + 8 = 18
d["Y"] = 20 (feuille)
Total = 18 + 20 = 38

Mini-exercice B — Identifier le comportement bugué

Que renvoie la version buguée d'alerte_valeur_aberrante sur ce dictionnaire avec limite = 100 ? 

{"A": {"x": 50, "y": 80}, "B": {"z": 200}}

La version buguée examine "A" en premier. "A" est un dict → return alerte_valeur_aberrante({"x":50,"y":80}, 100). 50 ≤ 100, 80 ≤ 100 → renvoie False. La boucle s'arrête — "B" n'est jamais examiné.

Résultat bugué : False. Résultat correct : True (car 200 > 100).

Mini-exercice C — Choix de total_simple vs total_rec

Pour chacun de ces dictionnaires, laquelle des deux fonctions (total_simple ou total_rec) fonctionne correctement ?

  1. {"a": 10, "b": 20}
  2. {"a": {"b": 10}, "c": 20}
  3. {"a": {"b": {"c": 5}}}
  1. Les deux fonctionnent — pas de sous-dictionnaire.
  2. Seule total_rec fonctionne — total_simple essaierait d'additionner le dictionnaire {"b": 10} à 20, ce qui provoquerait une TypeError.
  3. Seule total_rec fonctionne — imbrication de profondeur 3.
Récapitulatif — Ce que tu dois retenir
  • dico.values() donne accès aux valeurs ; dico.items() donne les paires clé/valeur.
  • La récursivité sur dictionnaires imbriqués suit toujours le même schéma : tester si la valeur est un dict, et si oui, rappeler la fonction.
  • Dans un appel récursif booléen, ne jamais écrire return appel_recursif(...) dans une boucle — utiliser if appel_recursif(...): return True.
  • Un bon jeu de tests couvre : le cas vide, le cas trivial, la valeur exactement égale à la limite, la valeur dans la dernière branche, et l'imbrication profonde.
  • isinstance(objet, dict) est le test fiable pour savoir si une valeur est un dictionnaire.
