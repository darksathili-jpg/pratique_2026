1 — Rappels : Programmation Orientée Objet

Tu connais déjà les listes et les dictionnaires. Mais quand un objet du monde réel possède à la fois des données (les fruits disponibles, la carte des recettes) et des comportements (vérifier si un smoothie est faisable, lister les alternatives), la POO permet de tout regrouper proprement dans une seule classe.

Rappel — Classe et méthode

Une classe est un modèle qui regroupe des données (attributs) et des fonctions (méthodes) qui opèrent sur ces données. Une instance est un objet concret créé d'après ce modèle.

Le premier paramètre de chaque méthode est toujours self — il désigne l'instance elle-même et permet d'accéder à ses attributs depuis l'intérieur de la classe.

Exemple minimal 
class Panier:
    def __init__(self, fruits):
        self.fruits = fruits          # attribut de l'instance

    def contient(self, fruit):        # méthode
        return fruit in self.fruits   # accès via self

p = Panier(["Mangue", "Kiwi"])
print(p.contient("Kiwi"))   # True
print(p.contient("Fraise")) # False

Tester l'appartenance à une liste

Dans ce sujet, tu auras souvent besoin de vérifier si un fruit est présent dans la liste des fruits disponibles. Python offre une syntaxe très lisible pour ça :

L'opérateur in 
fruits_dispo = ["Mangue", "Ananas", "Banane"]

"Mangue" in fruits_dispo   # True
"Kiwi"   in fruits_dispo   # False

Pour vérifier que tous les éléments d'une liste sont dans une autre, la fonction all() combinée à une expression génératrice est très pratique : 

recette = ["Mangue", "Ananas", "Banane"]
dispo   = ["Mangue", "Ananas", "Banane", "Kiwi"]

all(fruit in dispo for fruit in recette)   # True — tous les fruits sont là

Compter les éléments communs entre deux listes

Pour la notion d'alternative (Questions 3 et 4), tu devras compter combien de fruits deux recettes ont en commun. La boucle suivante est le schéma de base : 

def nb_communs(liste1, liste2):
    compteur = 0
    for element in liste1:
        if element in liste2:
            compteur += 1
    return compteur
Astuce — Version avec sum() 
sum(1 for f in liste1 if f in liste2)  # identique, plus compact

Les deux approches sont correctes — utilise celle que tu trouves la plus lisible.

Question 1 — smoothie_possible

Énoncé — Question 1

Q1 Écrire la méthode smoothie_possible(self, nom_smoothie) qui prend en paramètre le nom d'un smoothie et renvoie True si ce smoothie peut être préparé avec les fruits disponibles, False sinon.

Un smoothie est possible si et seulement si chacun de ses fruits figure dans self.liste_fruits_disponibles.

Indice 1

La liste des fruits du smoothie nom_smoothie se trouve dans self.db_smoothies[nom_smoothie]. Par exemple, pour « Tropical » : ["Mangue", "Ananas", "Banane"]. La liste des fruits disponibles est dans self.liste_fruits_disponibles.

Indice 2

Deux approches équivalentes :

# Approche 1 : boucle avec retour anticipé
for fruit in recette:
    if fruit not in self.liste_fruits_disponibles:
        return False   # un fruit manque → impossible
return True            # tous les fruits trouvés

# Approche 2 : all() en une ligne
return all(fruit in self.liste_fruits_disponibles
           for fruit in recette)
Éditeur Python — Complète la méthode
Correction — Question 1

Logique : dès qu'un fruit de la recette est absent de liste_fruits_disponibles, on renvoie False immédiatement (retour anticipé — inutile de continuer). Si la boucle se termine sans trouver de fruit manquant, c'est que tous sont présents : on renvoie True.

Question 2 — liste_smoothies_possibles

Énoncé — Question 2

Q2 Écrire la méthode liste_smoothies_possibles(self) qui ne prend pas de paramètre (hormis self) et renvoie la liste de tous les noms de smoothies réalisables avec les fruits disponibles.

Tu peux — et tu dois — réutiliser la méthode smoothie_possible que tu viens d'écrire.

Indice 1

Tous les noms de smoothies sont les clés de self.db_smoothies. Pour les parcourir : for nom in self.db_smoothies: (ou self.db_smoothies.keys(), c'est équivalent).

Indice 2

Schéma classique de filtrage :

possibles = []
for nom in self.db_smoothies:
    if self.smoothie_possible(nom):
        possibles.append(nom)
return possibles

Version compacte avec compréhension de liste :

return [nom for nom in self.db_smoothies
        if self.smoothie_possible(nom)]
Éditeur Python — Complète la méthode
Correction — Question 2

Réutilisation : la méthode délègue entièrement le test à smoothie_possible. C'est le principe fondamental de la POO : chaque méthode a une responsabilité précise, et les autres s'appuient dessus. Si tu corriges smoothie_possible plus tard, liste_smoothies_possibles en bénéficie automatiquement.

Question 3 — Tester score_proximite

Énoncé — Question 3

Q3 La méthode score_proximite(self, nom1, nom2) est fournie : elle renvoie le nombre de fruits communs entre deux smoothies. Compléter la fonction test_score_proximite avec des tests pertinents.

Voici le code de la méthode fournie :

def score_proximite(self, nom1, nom2):
    """Renvoie le nombre de fruits communs entre deux smoothies."""
    nb = 0
    fruits1 = self.db_smoothies[nom1]
    fruits2 = self.db_smoothies[nom2]
    for fruit in fruits1:
        if fruit in fruits2:
            nb += 1
    return nb

Tropical (Mangue, Ananas, Banane) Tropical (Mangue, Ananas, Banane) tous 3

Correction — Question 3

Propriété utile : le score est commutatif — score(A, B) == score(B, A) — car on compte les fruits en commun, indépendamment de l'ordre. C'est un bon test supplémentaire à inclure systématiquement.

Question 4 — Corriger plus_proche_possible

Énoncé — Question 4

Q4 Observer avec test_plus_proche_possible que le smoothie renvoyé n'est pas le bon. Identifier le problème dans plus_proche_possible, proposer une démarche de résolution et la mettre en œuvre.

Voici le code fourni :

def plus_proche_possible(self, nom_smoothie_ref):
    max_communs = 0
    smoothie_proche = None
    for nom_smoothie in self.db_smoothies:
        nb_communs = self.score_proximite(nom_smoothie_ref, nom_smoothie)
        if nb_communs > max_communs:
            max_communs = nb_communs
            smoothie_proche = nom_smoothie
    return smoothie_proche

Le test attendu : pour la boutique avec Mangue, Ananas, Banane, Fraise, Citron, Kiwi et Pomme verte, le smoothie le plus proche du « Tropical » (Mangue, Ananas, Banane) parmi les smoothies possibles devrait être « Tropical citron » (Mangue, Ananas, Citron) — 2 fruits en commun. Mais la fonction renvoie « Tropical » lui-même.

Indice 1

score_proximite("Tropical", "Tropical") renvoie 3 (un smoothie partage 3 fruits avec lui-même). La boucle trouve donc « Tropical » avec le score maximal 3 — et le renvoie, même si « Tropical » n'est pas possible ! La fonction cherche le plus proche parmi tous les smoothies, alors qu'elle devrait chercher parmi les smoothies possibles uniquement.

Indice 2

Il suffit de remplacer le parcours de self.db_smoothies par un parcours de self.liste_smoothies_possibles() : 

# Avant (bugué) :
for nom_smoothie in self.db_smoothies:

# Après (corrigé) :
for nom_smoothie in self.liste_smoothies_possibles():

Ainsi, le smoothie de référence lui-même n'est candidat que s'il est réalisable — ce qui est cohérent avec l'objectif de la fonctionnalité « alternative ».

Smoothies possibles : Tropical, Tropical citron, Vert citron, Rouge kiwi.

La correction doit aussi exclure le smoothie de référence lui-même de la liste des candidats — on cherche une alternative, pas le smoothie original.

Éditeur Python — Corrige la méthode
Correction — Question 4

Les deux corrections nécessaires :

  1. Remplacer self.db_smoothies par self.liste_smoothies_possibles() — on ne propose comme alternative que des smoothies effectivement réalisables.
  2. Exclure le smoothie de référence lui-même avec if nom_smoothie == nom_smoothie_ref: continue — si le smoothie original est possible, il ne doit pas être sa propre « alternative ».

Cas None : si aucun smoothie possible ne partage de fruit avec le smoothie de référence, max_communs reste à 0 et smoothie_proche reste à None. La fonction renvoie correctement None dans ce cas.

Pièges fréquents & astuces de vérification

Piège 1 — Oublier self dans les méthodes

Toute méthode d'une classe doit avoir self comme premier paramètre. Pour accéder aux attributs depuis l'intérieur de la classe, il faut toujours écrire self.attribut — jamais juste attribut. Et pour appeler une autre méthode depuis la même classe : self.smoothie_possible(nom), pas juste smoothie_possible(nom).

Piège 2 — Chercher dans tous les smoothies plutôt que dans les possibles

C'est exactement le bug de la Question 4. Si tu cherches le « plus proche » parmi self.db_smoothies, tu inclus des smoothies non réalisables — et le smoothie de référence lui-même (score 3 avec lui-même, imbattable). Il faut toujours restreindre la recherche à self.liste_smoothies_possibles().

Piège 3 — Retourner True trop tôt dans smoothie_possible

Le return True ne doit être atteint qu'après avoir parcouru toute la recette sans trouver de fruit manquant. Placer return True à l'intérieur de la boucle provoquerait un retour dès le premier fruit trouvé — ce qui ne garantit pas que les autres sont présents.

Piège 4 — Ne pas gérer le cas None dans affichage_possibles

Quand aucun smoothie possible ne partage de fruit avec un smoothie demandé, plus_proche_possible renvoie None. Il faut tester if proche is not None: avant d'afficher l'alternative — sous peine d'afficher « essayez None » au client, ce qui n'est pas très élégant.

Récapitulatif — Ce que tu dois retenir
  • Dans une méthode, self est obligatoire en premier paramètre — il donne accès aux attributs et aux autres méthodes.
  • element in liste renvoie True si l'élément est présent dans la liste.
  • all(condition for x in liste) teste si la condition est vraie pour tous les éléments.
  • Le bug de plus_proche_possible : chercher dans tous les smoothies au lieu des smoothies possibles uniquement.
  • Ne jamais oublier d'exclure le smoothie de référence lui-même de la recherche d'alternative.
  • Toujours gérer le cas None quand une fonction peut ne rien trouver.
