1 — Rappels : outils Python du sujet

Listes de dictionnaires — filtrer et accumuler

Chaque relevé de température est un dictionnaire. La liste entière est un tableau de mesures. Pour ne travailler que sur une zone, on filtre d'abord.

Patron universel – filtrer puis calculer 
# Étape 1 : garder uniquement les relevés de la zone voulue
releves_zone = [r for r in donnees if r['zone'] == zone]

# Étape 2 : vérifier qu'il y a des données
if not releves_zone:          # liste vide → renvoyer None ou {}
    return None

# Étape 3 : extraire les températures et calculer
temperatures = [r['temperature'] for r in releves_zone]
moyenne = sum(temperatures) / len(temperatures)

Valeur absolue — abs()

Rappel – abs() en Python 
abs(3.0)    # → 3.0
abs(-3.0)   # → 3.0
abs(0)      # → 0

# Écart entre une valeur et une moyenne :
ecart = abs(temperature - moyenne)
# Anomalie si l'écart dépasse le seuil :
if ecart > seuil:
    anomalies.append(date)

Division entière — // et calcul de décennie

L'opérateur // (division entière) supprime la partie décimale. Pour obtenir la décennie d'une année, il faut diviser par 10 puis multiplier par 10 — deux opérations, pas une seule.

Question 1 — Écrire temperature_moyenne

Énoncé — Question 1

Q1 Écrire temperature_moyenne(zone, donnees) qui renvoie la température moyenne (float) de tous les relevés de la zone, ou None si aucun relevé n'existe pour cette zone. 

>>> temperature_moyenne('Societe', donnees_test)
27.75
>>> temperature_moyenne('Australes', donnees_test)
>>>         # renvoie None

Aide méthodologique

Démarche pas à pas
  1. Filtrer : releves_zone = [r for r in donnees if r['zone'] == zone]
  2. Cas limite : si releves_zone est vide → return None
  3. Extraire les températures : [r['temperature'] for r in releves_zone]
  4. Calculer et renvoyer sum(temperatures) / len(temperatures)
Indice 1

Une liste en compréhension filtre et extrait en une seule ligne :
[r['temperature'] for r in donnees if r['zone'] == zone]
Si cette liste est vide (longueur 0), c'est que la zone n'existe pas → renvoyer None.

Indice 2 
def temperature_moyenne(zone, donnees):
    temperatures = [r['temperature'] for r in donnees if r['zone'] == zone]
    if not temperatures:      # liste vide → zone inconnue
        return None
    return sum(temperatures) / len(temperatures)
Éditeur Python — Question 1
Correction — Question 1

La compréhension de liste fait d'un coup le filtrage et l'extraction. Pourquoi if not temperatures plutôt que if temperatures == [] ? En Python, une liste vide est falsy : not [] vaut True. Les deux formes fonctionnent, mais not temperatures est idiomatique.

Question 2 — Écrire detecter_anomalies

Énoncé — Question 2

Q2 Écrire detecter_anomalies(zone, seuil, donnees) qui renvoie la liste des dates dont la température s'écarte de la moyenne de plus de seuil degrés (en valeur absolue). Renvoyer une liste vide si la zone n'existe pas.

Aide méthodologique

Démarche
  1. Calculer la moyenne avec temperature_moyenne (déjà écrite en Q1).
  2. Si la moyenne est None → renvoyer [].
  3. Parcourir les relevés de la zone.
  4. Pour chaque relevé : si abs(r['temperature'] - moyenne) > seuil, ajouter r['date'] à la liste résultat.
  5. Renvoyer la liste.
Exemple chiffré — Société, seuil = 1.0, moy = 27.75 
abs(26.5 - 27.75) = 1.25 > 1.0  → anomalie  → date '2010-06-20' ajoutée
abs(27.0 - 27.75) = 0.75 ≤ 1.0  → normale
abs(27.5 - 27.75) = 0.25 ≤ 1.0  → normale
abs(28.0 - 27.75) = 0.25 ≤ 1.0  → normale
abs(28.5 - 27.75) = 0.75 ≤ 1.0  → normale
abs(29.0 - 27.75) = 1.25 > 1.0  → anomalie  → date '2021-05-30' ajoutée

Résultat : ['2010-06-20', '2021-05-30']

Indice 1

temperature_moyenne renvoie None pour une zone inconnue. Tester ce cas en premier : if moyenne is None: return []. Ne pas utiliser if moyenne == None — préférer is None.

Indice 2 
def detecter_anomalies(zone, seuil, donnees):
    moyenne = temperature_moyenne(zone, donnees)
    if moyenne is None:
        return []
    anomalies = []
    for r in donnees:
        if r['zone'] == zone:
            if abs(r['temperature'] - moyenne) > seuil:
                anomalies.append(r['date'])
    return anomalies
Éditeur Python — Question 2
Correction — Question 2

Réutiliser temperature_moyenne : ne pas recalculer la moyenne — appeler la fonction déjà écrite. C'est le principe DRY (Don't Repeat Yourself). Si la Q1 contient une erreur, la corriger là corrige aussi la Q2 automatiquement.

Ordre des relevés dans le résultat : puisqu'on parcourt donnees dans l'ordre, les dates sont renvoyées dans l'ordre d'apparition dans la liste — ce qui correspond à l'ordre chronologique si les données sont triées.

Question 3 — Écrire les trois fonctions de test

Énoncé — Question 3

Q3 Compléter les trois fonctions de test pour evolution_par_decennie. Tes tests doivent permettre d'identifier le bug présent dans le code.

  1. test_zone_inexistante() — zone qui n'existe pas → dictionnaire vide
  2. test_une_seule_decennie() — zone Marquises → une seule décennie 2020
  3. test_plusieurs_decennies() — zone Société → décennies 2010 et 2020

Comment des tests révèlent un bug ?

Le bug change les clés du dictionnaire (201 au lieu de 2010). Un test qui vérifie que la clé 2010 est présente dans le résultat va donc échouer avec la version buguée — c'est exactement ce qu'on veut.

Pourquoi le test 3 révèle le bug

Avec le bug : evolution_par_decennie('Societe', donnees_test) renvoie {201: 27.0, 202: 28.5}.
Le test assert 2010 in resultat échoue car la clé est 201, pas 2010.
Sans ce test, le bug passerait inaperçu si on ne vérifie que les valeurs (les moyennes sont correctes !) — seules les clés sont fausses.

Correction — Question 3

Pourquoi les messages d'assertion sont importants : écrire assert 2010 in resultat, f"Clé 2020 absente (clés présentes : {list(resultat.keys())})" au lieu d'un simple assert 2010 in resultat permet de voir immédiatement quelles clés ont été générées — ici [201, 202] — ce qui pointe directement vers le bug.

Pourquoi le test 1 ne révèle pas le bug : la zone inexistante renvoie {} que le code soit bugué ou non. C'est le test 3 qui est décisif.

Question 4 — Identifier et corriger le bug

Énoncé — Question 4

Q4 Après avoir identifié le bug grâce aux tests, corriger evolution_par_decennie.

Identification précise du bug

Le bug — ligne 8 de evolution_par_decennie 
# LIGNE BUGUÉE :
decennie = (annee // 10)

# Résultat pour chaque année :
# 2010 // 10 = 201   ← devrait être 2010
# 2015 // 10 = 201   ← devrait être 2010
# 2020 // 10 = 202   ← devrait être 2020
# 2023 // 10 = 202   ← devrait être 2020

// donne le quotient de la division par 10 — soit le nombre de décennies écoulées depuis l'an 0. Pour obtenir la décennie en format année, il faut remettre à l'échelle en multipliant par 10.

Correction — une seule ligne à modifier 
# AVANT (bugué) :
decennie = (annee // 10)

# APRÈS (corrigé) :
decennie = (annee // 10) * 10

# Vérification :
# (2010 // 10) * 10 = 201 * 10 = 2010  ✓
# (2015 // 10) * 10 = 201 * 10 = 2010  ✓ (groupé avec 2010)
# (2020 // 10) * 10 = 202 * 10 = 2020  ✓
# (2023 // 10) * 10 = 202 * 10 = 2020  ✓
Résultats avant/après correction — Société

Buguée{201: 27.0, 202: 28.5}❌ 201, 202

Note : les valeurs des moyennes sont identiques dans les deux versions — seules les clés changent. C'est pourquoi un test qui ne vérifie que les valeurs et non les clés ne détecterait pas ce bug.

Éditeur Python — Question 4 (version corrigée)
Correction — Question 4

La correction se résume à ajouter * 10 à la ligne du calcul de la décennie :

# Avant (bugué) :
decennie = (annee // 10)

# Après (corrigé) :
decennie = (annee // 10) * 10

Explication mathématique : annee // 10 est la division euclidienne de l'année par 10. Elle donne le quotient — le nombre de décennies depuis l'an 0. Pour retrouver l'année de début de la décennie, on multiplie ce quotient par 10. C'est l'opération inverse : \((annee \div 10) \times 10 = \lfloor annee / 10 \rfloor \times 10\).

Propriété utile : cette formule s'applique à n'importe quelle granularité temporelle. Pour arrondir au siècle : (annee // 100) * 100. Pour arrondir à la quinzaine : (valeur // 15) * 15.

Pièges fréquents & astuces de vérification

Piège 1 — Oublier le * 10 dans le calcul de décennie

annee // 10 donne le quotient (201 ou 202), pas la décennie (2010 ou 2020). Le résultat ne lève pas d'erreur Python — les moyennes sont calculées, juste sous des clés incorrectes. Sans tester les clés du dictionnaire, ce bug est invisible.

Piège 2 — Tester seulement les valeurs et pas les clés

Le bug de ce sujet ne modifie pas les valeurs des moyennes — il ne change que les clés. assert resultat[201] == 27.0 passerait avec la version buguée ! Seul assert 2010 in resultat révèle le problème. Un test efficace vérifie à la fois les clés et les valeurs.

Piège 3 — Utiliser == None au lieu de is None

En Python, is None est idiomatique pour tester l'absence de valeur. == None fonctionne aussi mais est déconseillé car il peut être trompé par des objets qui redéfinissent __eq__. Les tests du BAC utilisent souvent is None dans les assertions.

Piège 4 — Oublier int() sur le résultat de split

date.split('-')[0] renvoie la chaîne "2020", pas l'entier 2020. Sans int(...), la division entière // 10 lève une TypeError. Toujours convertir : annee = int(date.split('-')[0]).

Piège 5 — Comparer des flottants avec ==

assert temperature_moyenne('Tuamotu', donnees_test) == 27.43 peut échouer à cause des erreurs d'arrondi flottant. Pour les moyennes non-entières, préférer : assert abs(resultat - 27.433) < 0.001. En revanche, 27.0, 28.5, 26.0 sont des sommes de nombres avec des représentations binaires exactes — on peut les comparer avec ==.

Astuces de vérification
  • Vérifier les clés ET les valeurs : pour un dictionnaire de résultats, toujours tester assert clé in resultat avant assert resultat[clé] == valeur_attendue — sinon un KeyError masque le vrai problème.
  • Test de non-régression : après correction, relancer tous les tests (Q3 + Q4) pour s'assurer que la correction ne casse rien.
  • Invariant de taille : len(detecter_anomalies(zone, seuil_tres_grand, donnees)) == 0 doit toujours valoir 0 (un seuil infini détecte zéro anomalie).
  • Invariant symétrique : detecter_anomalies(zone, 0.0, donnees) doit renvoyer toutes les dates de la zone (écart > 0 est toujours vrai sauf si la valeur est exactement la moyenne).
