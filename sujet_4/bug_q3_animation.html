Question 3 — Le bug du parcours modifié

Modifier une liste pendant qu'on la parcourt avec for — animation interactive

01 Le code buggué
Code fourni dans culture.py — VERSION BUGUÉE 
def purger_mesures_extremes(liste_mesures):
    for mesure in liste_mesures:          # parcours direct de la liste
        if mesure['temperature'] < 20 or mesure['temperature'] > 25:
            liste_mesures.remove(mesure)  # ← BUG : on modifie pendant le parcours !
    return liste_mesures

La fonction est censée ne garder que les mesures entre 20°C et 25°C. Elle supprime les extrêmes avec .remove() pendant le parcours for. Résultat : certaines mesures extrêmes survivent sans que Python ne lève la moindre erreur.

03 La mécanique du bug

La boucle for mesure in liste_mesures maintient un compteur d'indice interne que Python incrémente automatiquement à chaque tour. Ce compteur est aveugle aux suppressions : il avance de 1 quoi qu'il arrive.

Mécanisme — Décalage après suppression 
Liste initiale  :  [18.0, 19.0, 22.0, 28.0, 29.0]
Indices         :   [  0  ,   1  ,   2  ,   3  ,   4  ]

── Itération 1 : indice = 0 ──
Lecture → 18.0 (extrême)
.remove(18.0) → la liste se compacte :
Après           :  [19.0, 22.0, 28.0, 29.0]
Indices         :   [ 0  ,   1  ,   2  ,   3  ]

── Itération 2 : indice = 1 ──       ← Python avance à 1
⚠ 19.0 est maintenant à l'indice 0 — Python le SAUTE !
Lecture → 22.0 (dans la plage, conservé)

── Itération 3 : indice = 2 ──
Lecture → 28.0 (extrême)
.remove(28.0) → compactage :
Après           :  [19.0, 22.0, 29.0]
                                         ↑ passé de l'indice 3 à l'indice 2

── Itération 4 : indice = 3 ──       ← hors bornes → fin de boucle
29.0 n'est jamais examiné !

Résultat retourné : [19.0, 22.0, 29.0]  — 19.0 et 29.0 auraient dû être supprimés

Règle absolue en Python :
ne jamais supprimer ou ajouter des éléments dans une liste
pendant qu'on la parcourt avec for

Aucune exception n'est levée — le bug est silencieux et le résultat est incorrect.

04 La correction (Q4)

La solution idiomatique en Python est la compréhension de liste : au lieu de modifier la liste en place, on en crée une nouvelle contenant uniquement les éléments à garder.

Version buguée 
def purger(liste_mesures):
    for m in liste_mesures:
        if m['temperature'] < 20 \
        or m['temperature'] > 25:
            liste_mesures.remove(m)
    return liste_mesures
Version corrigée 
def purger(liste_mesures):
    return [
        m for m in liste_mesures
        if 20 <= m['temperature'] <= 25
    ]

# La liste originale n'est pas touchée.
# On garde ce qui est dans la plage.
Trace d'exécution — Version corrigée sur [18.0, 19.0, 22.0, 28.0, 29.0] 
Parcours de la liste ORIGINALE (inchangée) :
  18.0 → hors plage → non inclus
  19.0 → hors plage → non inclus
  22.0 → dans [20,25] → inclus
  28.0 → hors plage → non inclus
  29.0 → hors plage → non inclus   ← jamais sauté !

Résultat : [ {'jour': 3, 'temperature': 22.0, ...} ]
05 À retenir
Pourquoi ça arrive

La boucle for utilise un compteur d'indice interne. Quand un élément est supprimé, tous les suivants reculent d'un cran. L'indice avance quand même — l'élément qui a reculé est sauté.

Pourquoi c'est dangereux

Python ne lève aucune exception. Le code s'exécute jusqu'à la fin sans message d'erreur. Le bug est silencieux : seul un test attentif le révèle.

La solution

Utiliser une compréhension de liste : [x for x in l if condition]. On crée une nouvelle liste — l'originale n'est jamais modifiée pendant le parcours.
