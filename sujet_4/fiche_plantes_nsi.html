1 — Rappels : la Programmation Orientée Objet

Tu as déjà manipulé des listes, des dictionnaires, des entiers. Ces types de données sont fournis par Python. Mais parfois, on a besoin de représenter des entités plus complexes — comme une plante, avec son nom, son espèce, sa taille, etc. La POO permet de créer ses propres types de données sur mesure.

Définition — Classe, instance, attribut

Une classe est un modèle (ou « moule ») qui décrit la structure d'un objet. Une instance est un objet concret créé d'après ce modèle. Les attributs sont les données propres à chaque instance.

La méthode spéciale __init__ est appelée automatiquement à la création d'une instance : c'est le constructeur. Le paramètre self désigne toujours l'instance en cours de création.

Voici la classe Plante du sujet :

class Plante:
    def __init__(self, nom, espece, croissance, taille, exposition):
        self.nom        = nom         # chaîne : nom courant
        self.espece     = espece      # chaîne : nom latin
        self.croissance = croissance  # entier : durée en jours
        self.taille     = taille      # entier : hauteur maximale en cm
        self.exposition = exposition  # chaîne : "ombre", "mi-ombre" ou "plein soleil"
Exemple concret — Créer et interroger une instance 
# Création d'une instance
basilic = Plante("Basilic", "Ocimum basilicum", 60, 40, "plein soleil")

# Accès aux attributs avec la notation pointée
print(basilic.nom)         # affiche : Basilic
print(basilic.croissance)  # affiche : 60
print(basilic.taille)      # affiche : 40

Retiens bien la notation pointée : objet.attribut. C'est la seule façon d'accéder aux données d'une instance depuis l'extérieur de la classe.

Parcourir une liste d'instances

Dans ce sujet, les plantes sont stockées dans une liste Python ordinaire. Pour accéder aux attributs de chaque plante, on combine la boucle for avec la notation pointée : 

for p in plantes:
    print(p.nom, "→", p.croissance, "jours")
Rappel — Moyenne d'une liste de valeurs

La moyenne de \(n\) valeurs \(v_1, v_2, \ldots, v_n\) est : $$\bar{v} = \frac{v_1 + v_2 + \cdots + v_n}{n} = \frac{\displaystyle\sum_{i=1}^{n} v_i}{n}$$

En Python : sum(liste) / len(liste) — mais attention à ne pas diviser par zéro si la liste est vide.

Question 1 — Moyenne de croissance

Énoncé — Question 1

Q1 Écrire une fonction croissance_moyenne(plantes) qui prend en paramètre une liste d'instances de la classe Plante et renvoie la moyenne des durées de croissance (attribut croissance) de l'ensemble des plantes.

Si la liste fournie est vide, la fonction doit renvoyer None. Écrire au moins deux tests, dont un traitant le cas d'une liste vide.

Indice 1

La première chose à faire dans la fonction est de tester si la liste est vide. En Python, une liste vide est falsy : if not plantes: est équivalent à if len(plantes) == 0:. Si la liste est vide, retourne immédiatement None.

Indice 2

La durée de croissance de chaque plante est dans l'attribut croissance. Pour calculer la somme de toutes les durées avec une compréhension de liste : 

total = sum(p.croissance for p in plantes)

Il ne reste plus qu'à diviser par len(plantes).

Éditeur Python — Complète la fonction
Correction — Question 1

Explication : le garde if not plantes intercepte le cas vide avant tout calcul, évitant une division par zéro. L'expression génératrice p.croissance for p in plantes passe en revue tous les attributs croissance sans construire de liste intermédiaire. Le résultat est un float (division en Python 3).

Vérification manuelle : \(\frac{60 + 80 + 80 + 85 + 90}{5} = \frac{395}{5} = 79{,}0\).

Question 2 — Regrouper les mesures par plante

Énoncé — Question 2

Q2 Écrire une fonction dictionnaire_mesure(plantes, mesures) qui prend la liste des plantes et la liste des mesures, et renvoie un dictionnaire où :

  • chaque clé est le nom d'une plante présente dans plantes ;
  • chaque valeur est la liste de toutes les mesures concernant cette plante.

Si une plante n'a aucune mesure associée, sa liste doit être vide (pas absente). Concevoir une série de tests pertinente.

Format attendu en sortie 
{
  "Basilic": [
    {'jour': 1, 'plante': 'Basilic', 'hauteur': 0.85, ...},
    {'jour': 2, 'plante': 'Basilic', 'hauteur': 1.7,  ...},
    ...
  ],
  "Tomate":    [...],
  "Menthe":    [...],
  "Tournesol": [...],
  "Fougère":   []    # aucune mesure !
}
Indice 1

Commence par créer un dictionnaire avec une clé par plante, toutes initialisées à une liste vide. Cela garantit que même les plantes sans mesure apparaissent dans le résultat : 

resultat = {p.nom: [] for p in plantes}

Cette écriture compacte s'appelle une compréhension de dictionnaire.

Indice 2

Parcours ensuite toutes les mesures. Pour chaque mesure, le champ 'plante' donne le nom de la plante concernée. Vérifie que ce nom est une clé du dictionnaire, puis ajoute la mesure à la liste correspondante : 

for m in mesures:
    nom = m['plante']
    if nom in resultat:
        resultat[nom].append(m)

La condition if nom in resultat protège contre une mesure qui concernerait une plante inconnue (non présente dans la liste plantes).

Éditeur Python — Complète la fonction
Correction — Question 2

Logique en deux étapes : d'abord on prépare toutes les clés (une par plante, avec liste vide), puis on distribue les mesures. Cette séparation garantit qu'aucune plante n'est oubliée dans le résultat.

Complexité : l'algorithme est en \(\mathcal{O}(|\text{plantes}| + |\text{mesures}|)\) — on parcourt chaque liste exactement une fois.

Question 3 — Analyser le bug

Énoncé — Question 3

Q3 La fonction purger_mesures_extremes ci-dessous est censée retirer de la liste toutes les mesures dont la température est strictement inférieure à 20°C ou strictement supérieure à 25°C. Exécuter la fonction de test et analyser son code pour identifier la source de l'erreur logique.

Voici le code fourni dans culture.py :

def purger_mesures_extremes(liste_mesures):
    for mesure in liste_mesures:
        if mesure['temperature'] < 20 or mesure['temperature'] > 25:
            liste_mesures.remove(mesure)  # ← BUG ici
    return liste_mesures


def test_purger():
    mesures_test = [
        {'jour': 1, 'plante': 'Basilic', 'temperature': 18.0},
        {'jour': 2, 'plante': 'Basilic', 'temperature': 19.0},
        {'jour': 3, 'plante': 'Basilic', 'temperature': 22.0},
        {'jour': 4, 'plante': 'Basilic', 'temperature': 28.0},
        {'jour': 5, 'plante': 'Basilic', 'temperature': 29.0},
    ]
    purger_mesures_extremes(mesures_test)
    for m in mesures_test:
        print(f"Jour {m['jour']} : {m['temperature']}°C")

En exécutant ce test, tu remarqueras que la mesure du jour 2 (19°C) est encore présente dans la liste après la purge. Pourquoi ?

Indice 1

La boucle for mesure in liste_mesures utilise un indice interne qui avance de 1 à chaque itération, même si des éléments ont été supprimés de la liste entre-temps.

Trace mentalement ce qui se passe quand on supprime l'élément à l'indice 0 : tous les éléments suivants se décalent d'une position vers la gauche. L'indice interne passe ensuite à 1 — mais l'ancien élément 1 est maintenant à l'indice 0. Il est donc sauté.

Indice 2 — Trace d'exécution

Liste de départ (indices 0 à 4) : [18.0, 19.0, 22.0, 28.0, 29.0]

ItérationIndice interneÉlément luActionListe après
1018.0 → extrêmeSuppression[19.0, 22.0, 28.0, 29.0]
2122.0 (saut !)Conservé[19.0, 22.0, 28.0, 29.0]
3228.0 → extrêmeSuppression[19.0, 22.0, 29.0]
43hors bornesFin de boucle[19.0, 22.0, 29.0]

Le 19.0 et le 29.0 survivent alors qu'ils devraient être supprimés. C'est l'effet classique du décalage d'indices lors d'une modification d'une liste pendant son parcours.

Éditeur Python — Exécute le test pour observer le bug
Correction — Question 3

Source du bug : la fonction modifie la liste liste_mesures pendant qu'elle la parcourt avec for mesure in liste_mesures.

En Python, la boucle for sur une liste utilise un compteur d'indice interne qui s'incrémente de 1 à chaque tour. Quand on supprime un élément avec .remove(), les éléments suivants se décalent d'un cran vers la gauche. Le compteur avance quand même, ce qui fait sauter l'élément qui vient de prendre la place de l'élément supprimé.

Concrètement : après suppression du jour 1 (18°C), le jour 2 (19°C) passe à l'indice 0. La boucle avance à l'indice 1 et lit le jour 3 (22°C) — le jour 2 n'est jamais examiné et reste dans la liste.

Règle à retenir : on ne doit jamais modifier une liste pendant qu'on la parcourt avec for.

Question 4 — Corriger la fonction de purge

Énoncé — Question 4

Q4 Proposer une version corrigée de purger_mesures_extremes qui supprime réellement toutes les mesures hors plage [20°C, 25°C].

Rappel — Stratégies classiques

Trois approches permettent d'éviter le bug de modification pendant le parcours :

  • Construire une nouvelle liste en gardant uniquement les éléments conformes (compréhension de liste ou boucle avec append). C'est l'approche la plus lisible.
  • Parcourir une copie de la liste avec for m in liste_mesures[:] et supprimer les éléments indésirables de la liste originale.
  • Parcourir à rebours (range(len(L)-1, -1, -1)) pour que les suppressions ne décalent que des éléments déjà traités.

L'approche par nouvelle liste (filtrage) est la plus claire et la plus recommandée en Python.

Indice 1

Une compréhension de liste peut filtrer directement : 

valides = [m for m in liste_mesures if 20 <= m['temperature'] <= 25]

valides est une nouvelle liste ne contenant que les mesures dans la plage souhaitée. La liste originale n'est pas modifiée pendant la création de valides.

Indice 2

Si l'on souhaite que liste_mesures soit modifiée en place (sans en changer la référence), on peut vider la liste et la remplir avec les éléments filtrés : 

valides = [m for m in liste_mesures if 20 <= m['temperature'] <= 25]
liste_mesures.clear()
liste_mesures.extend(valides)

Ici, la fonction retourne la liste — mais elle a aussi modifié l'objet passé en paramètre, ce qui peut être utile si d'autres variables référencent la même liste.

Éditeur Python — Implémente la version corrigée
Correction — Question 4

Approche recommandée : la compréhension de liste (Approche 1) est la plus idiomatique en Python. Elle ne modifie pas la liste d'entrée — elle en crée une nouvelle, propre.

Point clé : la condition correcte est 20 <= m['temperature'] <= 25 (on garde les valeurs dans la plage), et non m['temperature'] < 20 or m['temperature'] > 25 (qui exprime les valeurs à rejeter). Les deux sont correctes si on adapte ce qu'on fait du résultat, mais la première est plus lisible dans un filtre.

Pourquoi l'Approche 2 est utile : si d'autres variables Python pointent sur le même objet liste, elles verront aussi la modification. Avec l'Approche 1, la liste originale n'est pas touchée.

Pièges fréquents & astuces de vérification

Piège 1 — Modifier une liste pendant son parcours

C'est précisément le bug de la Question 3. En Python, une boucle for sur une liste utilise un indice interne. Si on supprime des éléments pendant le parcours, des éléments sont sautés silencieusement. Aucune erreur n'est levée — le code tourne mais produit un résultat incorrect. Règle absolue : ne jamais appeler .remove(), .pop() ou del sur une liste pendant qu'on la parcourt avec for.

Piège 2 — Confondre attribut et clé de dictionnaire

Les plantes sont des instances de classe → on accède à leurs données avec la notation pointée : p.croissance, p.nom.
Les mesures sont des dictionnaires → on accède à leurs valeurs avec les crochets : m['temperature'], m['jour'].
Inverser les deux notations provoque une AttributeError ou une KeyError.

Piège 3 — Oublier la Fougère dans le dictionnaire

La Fougère est dans plantes mais n'a aucune mesure dans mesures. Si tu construis le dictionnaire en parcourant seulement les mesures, la Fougère sera absente du résultat — alors qu'elle doit y figurer avec une liste vide. C'est pourquoi l'initialisation {p.nom: [] for p in plantes} est indispensable.

Piège 4 — Division par zéro dans croissance_moyenne

Si tu oublies de tester le cas de la liste vide avant de calculer la moyenne, l'expression total / len(plantes) provoque une ZeroDivisionError. Le sujet impose explicitement de renvoyer None dans ce cas. Le test if not plantes: return None doit être la première ligne de la fonction.

Mini-exercices de vérification

Mini-exercice A — Calcul mental

Sans Python, calcule la moyenne des durées de croissance de ces trois plantes seulement : Basilic (60 j), Tomate (80 j), Tournesol (85 j).

\(\frac{60 + 80 + 85}{3} = \frac{225}{3} = 75{,}0\)

Mini-exercice B — Trace du bug

Trace pas à pas ce que fait la boucle buguée sur la liste [15.0, 22.0, 18.0, 24.0]. Quels éléments resteront-ils dans la liste après exécution ?

  • Itération 1 (indice 0) : 15.0 → extrême → supprimé. Liste : [22.0, 18.0, 24.0].
  • Itération 2 (indice 1) : 18.0 → extrême → supprimé. Liste : [22.0, 24.0]. Le 22.0 à l'indice 0 est sauté !
  • Itération 3 (indice 2) : hors bornes → fin de boucle.

Résultat : [22.0, 24.0]. Correct dans ce cas précis, mais le 22.0 n'a jamais été vérifié (il était dans la plage, donc ça passe). Le bug ne se voit pas toujours — c'est ce qui le rend dangereux.

Mini-exercice C — Notation pointée vs crochets

Parmi ces expressions, lesquelles sont correctes pour accéder à la température d'une mesure m et au nom d'une plante p ?

  1. m.temperature
  2. m['temperature']
  3. p['nom']
  4. p.nom

Correctes : 2 et 4.
m est un dictionnaire → notation crochets.
p est une instance de Plante → notation pointée.

Récapitulatif – Ce que tu dois retenir
  • Une instance de classe s'interroge avec la notation pointée (obj.attribut).
  • Un dictionnaire s'interroge avec les crochets (d['cle']).
  • Pour une moyenne, toujours tester la liste vide avant de diviser.
  • Pour grouper des données par catégorie, initialiser toutes les clés à [] avant de parcourir les données.
  • Ne jamais modifier une liste pendant qu'on la parcourt avec for — utiliser le filtrage par compréhension de liste à la place.
  • La bonne condition pour filtrer : 20 <= temp <= 25 (on garde ce qui est dans la plage).
