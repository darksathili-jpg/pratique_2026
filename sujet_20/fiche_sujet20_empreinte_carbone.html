1 — Rappels essentiels

Parcourir un dictionnaire

Il existe trois façons de parcourir un dictionnaire, selon ce dont tu as besoin : 

d = {'a': 10, 'b': 20}

for cle in d:               # parcourt les clés
    print(cle)              # → 'a', 'b'

for cle, valeur in d.items():  # parcourt paires (clé, valeur)
    print(cle, valeur)      # → 'a' 10  puis  'b' 20

for valeur in d.values():   # parcourt seulement les valeurs
    print(valeur)           # → 10, 20

Tester si une clé existe

if 'emails_simples' in utilisateur:
    print("L'utilisateur envoie des emails")

# Méthode .get() — valeur par défaut si clé absente
quantite = utilisateur.get('emails_simples', 0)  # 0 si absent
Clé absente ≠ valeur nulle

Si un utilisateur n'a pas la clé 'streaming_hd', ça ne veut pas dire qu'il fait 0 heure de streaming HD — ça veut dire qu'on n'a pas l'information. La convention dans ce sujet est : activité absente = 0 émission. utilisateur.get('streaming_hd', 0) gère ça proprement.

Construire un dictionnaire résultat

resultat = {'fort': [], 'moyen': [], 'faible': []}   # init avec listes vides

for activite in mon_dico:
    emission = mon_dico[activite] * EMISSIONS[activite]
    if emission >= 1000:
        resultat['fort'].append(activite)
    elif emission >= 200:
        resultat['moyen'].append(activite)
    else:
        resultat['faible'].append(activite)

La division par zéro — l'erreur de la Q4

10 / 0    # → ZeroDivisionError: division by zero

Cette erreur se produit quand le dénominateur vaut 0. Dans comparer_v2, le dénominateur est emission1. Si l'utilisateur 1 n'a pas l'activité, emission1 = 0 et la division explose. La correction consiste à tester avant de diviser.

5 — Q3 Compléter test_comparer

Énoncé de la question 3

Compléter la fonction test_comparer en ajoutant au moins deux tests pertinents qui vérifient différents aspects du comportement de la fonction. Pour chaque test, donner une brève justification. D'autres utilisateurs peuvent être créés.

Rappel : ce que fait comparer

comparer(u1, u2) renvoie un dictionnaire avec, pour chaque activité de EMISSIONS, la valeur émission_u2 − émission_u1. Si une activité est absente chez un utilisateur, son émission vaut 0.

Activité absente chez u2Vérifie la symétrie — absent = 0 dans les deux sensdiff = −émission_u1
Utilisateurs identiquesVérifie que la différence est 0 pour toutes les activitésToutes les valeurs = 0
Deux utilisateurs videsCas limite — aucune activité des deux côtésToutes les valeurs = 0

def test_comparer():
    # Test donné
    diff = comparer(utilisateur4, utilisateur5)
    assert diff['emails_simples'] == -200   # (50-100) × 4
    assert diff['recherches'] == 350        # (100-50) × 7

    # TEST 1 : activité absente chez u1 → traitée comme émission = 0
    ua = {'emails_simples': 0}
    ub = {'streaming_hd': 5}
    diff_ab = comparer(ua, ub)
    assert diff_ab['streaming_hd'] == 500   # (5-0) × 100
    # Justification : vérifie que l'absence d'activité chez u1 vaut bien 0

    # TEST 2 : utilisateurs identiques → toutes différences = 0
    diff_id = comparer(utilisateur4, utilisateur4)
    for activite in EMISSIONS:
        assert diff_id[activite] == 0
    # Justification : u1 == u2 → écart nul partout

    print("Les tests sont valides.")

6 — Q4 Identifier et corriger le bug de comparer_v2

Énoncé de la question 4

Dans certains cas, la fonction comparer_v2 provoque une erreur. Identifier dans quels cas l'erreur se produit et proposer une correction.

Le code fourni — version buguée

def comparer_v2(u1, u2):
    ecarts = {}
    for activite in EMISSIONS:
        quantite1 = 0
        quantite2 = 0
        if activite in u1:
            quantite1 = u1[activite]
        if activite in u2:
            quantite2 = u2[activite]
        emission1 = quantite1 * EMISSIONS[activite]
        emission2 = quantite2 * EMISSIONS[activite]
        ecarts[activite] = (emission2 - emission1) / emission1 * 100  # ← BUG si emission1 == 0
    return ecarts

Identification du bug

La formule \(\frac{emission_2 - emission_1}{emission_1} \times 100\) divise par emission1. Or si l'activité est absente chez l'utilisateur 1, quantite1 = 0 donc emission1 = 0 × EMISSIONS[activite] = 0. La division / 0 lève une ZeroDivisionError.

Cas qui provoquent l'erreur

L'erreur se produit quand l'utilisateur 1 n'a pas une activité (clé absente dans son dictionnaire), car cela donne emission1 = 0.

Exemple : comparer_v2(utilisateur4, utilisateur5) plante car utilisateur4 n'a pas 'emails_pj', 'streaming_sd', 'streaming_hd', ni 'stockage_cloud' → 4 divisions par zéro.

Correction

Avant de diviser, tester si emission1 == 0. Si oui, on ne peut pas calculer un pourcentage d'écart — on peut mettre None pour signaler « non applicable ».

Indice – la correction 
if emission1 == 0:
    ecarts[activite] = None   # impossible de calculer un % par rapport à 0
else:
    ecarts[activite] = (emission2 - emission1) / emission1 * 100

7 — Pièges fréquents & astuces de vérification

Piège 1 — Parcourir EMISSIONS au lieu de l'utilisateur (Q1)

Si tu parcours EMISSIONS au lieu de utilisateur, tu tentes d'accéder à utilisateur[activite] pour une activité absente → KeyError. Il faut parcourir l'utilisateur (ce qu'il a) et vérifier que chaque activité est dans EMISSIONS. 

# ❌ KeyError si l'utilisateur n'a pas toutes les activités
for activite in EMISSIONS:
    total += utilisateur[activite] * EMISSIONS[activite]

# ✓ Correct : parcourir l'utilisateur
for activite, quantite in utilisateur.items():
    if activite in EMISSIONS:
        total += quantite * EMISSIONS[activite]
Piège 2 — Oublier les listes vides dans classer_par_impact (Q2)

Si tu crées le dictionnaire résultat au fur et à mesure, tu risques de ne pas créer la clé 'faible' si aucune activité n'est faible. L'initialisation resultat = {'fort': [], 'moyen': [], 'faible': []} garantit que les trois clés existent toujours.

Piège 3 — Écrire elif emission > 200 au lieu de >= 200 (Q2)

Les seuils de l'énoncé sont inclusifs : 200 ≤ émissions < 1000. Une activité à exactement 200 gCO2e est de niveau moyen, pas faible. Utiliser > au lieu de >= classerait 200 en faible — faux. 

if emission >= 1000:          # ✓ ≥ 1000 → fort
elif emission >= 200:         # ✓ ≥ 200 → moyen
else:                         # ✓ < 200 → faible
Piège 4 — Ne pas justifier les tests dans test_comparer (Q3)

L'énoncé exige explicitement une brève justification pour chaque test ajouté. Un test sans commentaire sera pénalisé. Écris une ligne de commentaire (# Justification : ...) pour chaque assertion ajoutée.

Piège 5 — Corriger le bug de comparer_v2 en ignorant les cas None (Q4)

Certains élèves corrigent le bug en filtrant les activités absentes avec continue (ce qui fait que la clé n'existe pas dans le résultat). L'énoncé indique que la fonction traite toutes les activités de EMISSIONS. Il vaut mieux mettre None pour les cas non calculables plutôt que d'omettre la clé.

Astuce — Vérifier calculer_empreinte manuellement

Pour utilisateur2 (streaming_hd=15, emails_simples=100, recherches=10) : 15×100 + 100×4 + 10×7 = 1500 + 400 + 70 = 1970. Si ton résultat est différent, l'erreur est dans le parcours ou la multiplication.
