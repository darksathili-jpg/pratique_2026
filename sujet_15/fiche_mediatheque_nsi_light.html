1 — Rappels : chaînes, isdigit et SQLite

Parcourir une chaîne caractère par caractère

En Python, une chaîne de caractères est itérable : on peut la parcourir avec une boucle for exactement comme une liste. La méthode isdigit() renvoie True si et seulement si le caractère est un chiffre (0 à 9).

Exemple – filtrer les chiffres d'une chaîne 
# Garder uniquement les chiffres d'un ISBN brut
isbn_brut = "978-2-07-040 850-4"
resultat = ""
for c in isbn_brut:
    if c.isdigit():
        resultat += c
print(resultat)  # → "9782070408504"

# Version compacte (compréhension de chaîne)
resultat = "".join(c for c in isbn_brut if c.isdigit())
print(resultat)  # → "9782070408504"

Comparer des dates au format AAAAMMJJ

Les dates sont stockées au format "AAAAMMJJ" (ex. "20241015"). Ce format a une propriété précieuse : l'ordre alphabétique = l'ordre chronologique. On peut donc comparer des dates directement avec > ou <, sans conversion, en Python comme en SQL.

Exemple 
"20251201" > "20241015"   # True  → décembre 2025 après octobre 2024
"20240101" < "20240201"   # True  → janvier avant février
# En SQL :
WHERE emprunt.date_emprunt > '20240101'   # tous les emprunts après le 1er janv. 2024

Question 1 — Écrire normalisation_isbn

Énoncé — Question 1

Q1 Écrire la fonction normalisation_isbn(isbn) qui prend un numéro ISBN sous forme de chaîne quelconque (avec tirets, espaces, points…) et renvoie uniquement ses chiffres, dans l'ordre, sous forme de chaîne. 

normalisation_isbn("978-2-07-036024-5")   # → "9782070360245"
normalisation_isbn("978 2 07 040850 4")   # → "9782070408504"
normalisation_isbn("978.2.07.040850.4")   # → "9782070408504"
normalisation_isbn("9782253004226")        # → "9782253004226"  (déjà propre)
Contexte — Qu'est-ce qu'un ISBN ?

L'ISBN (International Standard Book Number) est le code-barre d'un livre. Le format actuel, ISBN-13, est composé de 13 chiffres groupés par tirets. Par exemple, 978-2-07-036024-5 est l'ISBN d'Un amour de Swann de Proust. Les trois groupements (978, 2-07-036024, 5) ont une signification (préfixe EAN, éditeur, titre, clé de contrôle), mais pour ce sujet on se contente d'extraire les 13 chiffres.

Aide méthodologique

Démarche — identique à normalisation_tel du sujet 15
  1. Initialiser une chaîne résultat vide : resultat = ""
  2. Parcourir chaque caractère c de isbn avec une boucle for.
  3. Si c.isdigit() est vrai, ajouter c à resultat.
  4. Renvoyer resultat.

def normalisation_isbn(isbn):

Question 2 — Jeu de tests de validation_isbn

Énoncé — Question 2

Q2 La fonction validation_isbn(isbn) est fournie et déjà correcte. Écrire un jeu de tests (assert) qui vérifie son bon fonctionnement en couvrant chaque condition de rejet indépendamment.

Rappel — logique de validation_isbn

def validation_isbn(isbn):
    """Renvoie True si isbn est un ISBN-13 valide (13 chiffres, commence par 978 ou 979)."""
    if len(isbn) != 13:               # doit avoir exactement 13 chiffres
        return False
    if not isbn.startswith("978") and not isbn.startswith("979"):
        return False                  # doit commencer par 978 ou 979
    return True

Question 3 — Écrire emprunts_livre_bd

Énoncé — Question 3

Q3 Écrire la fonction emprunts_livre_bd(conn, date) qui prend une connexion SQLite et une date (format "AAAAMMJJ"), et renvoie la liste des emprunts de livres de genre 'BD' effectués après cette date.

Chaque élément de la liste est un tuple contenant : (adherent.id, adherent.nom, adherent.prenom, adherent.telephone, emprunt.date_emprunt).

Les résultats doivent être triés par adherent.id puis par emprunt.date_emprunt.

Construction de la requête — étape par étape

Démarche — même structure que Q3 du sujet 15

def emprunts_livre_bd(conn, date):
    """
    Renvoie la liste des emprunts de BD effectués après `date`.
    Chaque tuple : (adherent.id, nom, prenom, telephone, date_emprunt).
    Résultats triés par adherent.id puis date_emprunt.
    """
    cursor = conn.cursor()
    resultats = cursor.execute(
        "SELECT adherent.id, adherent.nom, adherent.prenom, "
        "       adherent.telephone, emprunt.date_emprunt "
        "FROM emprunt "
        "JOIN livre    ON emprunt.id_livre    = livre.id "
        "JOIN adherent ON emprunt.id_adherent = adherent.id "
        "WHERE livre.genre = 'BD' "
        "AND   emprunt.date_emprunt > ? "
        "ORDER BY adherent.id, emprunt.date_emprunt",
        (date,)     # ← tuple à un élément : la virgule après date est OBLIGATOIRE
    )
    return list(resultats)

Points critiques à retenir :

  • Double JOIN : emprunt→livre (pour filtrer sur le genre) et emprunt→adhérent (pour récupérer les coordonnées).
  • Préfixer les colonnes : écrire adherent.id et non juste id pour lever toute ambiguïté (emprunt et adhérent ont tous les deux un champ id).
  • Paramètre (date,) : la virgule transforme date en tuple d'un élément — sans elle, Python passerait une chaîne et SQLite lèverait une erreur.
  • list(resultats) : convertir le curseur en liste pour pouvoir la réutiliser.

Énoncé — Question 4

Le code fourni (bugué)

Pièges fréquents & astuces de vérification

Piège 1 — Le tuple à un élément pour les paramètres SQL

cursor.execute("... WHERE ... > ?", date) envoie une chaîne comme paramètre — SQLite itère sur ses caractères et plante. Il faut impérativement un tuple, même pour un seul paramètre. La virgule après la valeur est obligatoire : (date,). 

# FAUX — date est une chaîne, pas un tuple
cursor.execute("... WHERE emprunt.date_emprunt > ?", date)

# JUSTE — tuple à un élément
cursor.execute("... WHERE emprunt.date_emprunt > ?", (date,))
Piège 2 — Oublier de convertir le curseur en liste

cursor.execute() renvoie un itérable paresseux, pas une liste. On ne peut le parcourir qu'une seule fois. Sans list(...), l'appelant peut se retrouver avec un objet vide dès le deuxième accès. 

# FAUX — renvoie un curseur
return cursor.execute("SELECT ...")

# JUSTE — convertit en liste réutilisable
return list(cursor.execute("SELECT ..."))
Piège 3 — Bug de dernier_emprunt : < vs >

Pour trouver le maximum d'une série (la date la plus récente), la condition de mise à jour est >. Pour le minimum, ce serait <. Confondre les deux produit un résultat plausible mais faux — c'est exactement le bug de ce sujet (et du sujet 15 officiel). 

# Trouver le maximum (date la plus récente)
if date > derniers[id_adh][4]:
    derniers[id_adh] = e   # on garde la plus grande
Piège 4 — Ambiguïté de colonnes dans le SELECT

Plusieurs tables ont des colonnes de même nom : emprunt.id et adherent.id sont deux colonnes id différentes. Sans préfixe (adherent.id, livre.genre…), SQLite peut renvoyer la mauvaise colonne. Toujours préfixer les colonnes dans les requêtes avec JOIN.

Piège 5 — Ne tester que les cas True dans validation_isbn

Un jeu de tests qui vérifie uniquement les ISBN valides (assert validation_isbn("9782070360245") == True) ne prouve rien sur les rejets. La fonction pourrait renvoyer True pour tout et les tests passeraient quand même. Il faut tester chaque condition de rejet avec un exemple précis.

Astuces de vérification
  • Pipeline complet : validation_isbn(normalisation_isbn(isbn_brut)). Par exemple, normalisation_isbn("978-2-07-036024-5") → "9782070360245" → validation_isbn("9782070360245") → True.
  • Tester dernier_emprunt avec un seul adhérent : une liste d'une seule entrée doit renvoyer un dictionnaire à une entrée, avec la date de cet unique emprunt — facile à vérifier à la main.
  • Vérifier la Q3 avec une date très ancienne : emprunts_livre_bd(conn, "19000101") doit renvoyer tous les emprunts de BD — si la liste est vide, la date ou le filtre genre est faux.
  • ISBN invalides courants : l'ancien format ISBN-10 (10 chiffres) est rejeté par validation_isbn car len != 13. C'est normal et attendu — ce sujet ne gère que l'ISBN-13.
