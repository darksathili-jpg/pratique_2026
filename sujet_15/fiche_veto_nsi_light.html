1 — Rappels : chaînes, isdigit et SQLite

Parcourir une chaîne caractère par caractère

En Python, une chaîne de caractères est itérable : on peut la parcourir avec une boucle for exactement comme une liste. La méthode isdigit() renvoie True si et seulement si le caractère est un chiffre (0 à 9).

Exemple – filtrer les chiffres d'une chaîne 
# Garder uniquement les chiffres d'une chaîne
chaine = "(0)6.12.99-90-12 gilbert"
resultat = ""
for c in chaine:
    if c.isdigit():
        resultat += c
print(resultat)  # → "0612999012"

# Version compacte (compréhension de chaîne)
resultat = "".join(c for c in chaine if c.isdigit())
print(resultat)  # → "0612999012"

Requêtes SQL avec sqlite3 en Python

Patron complet – connexion, requête, résultats 
import sqlite3

conn = sqlite3.connect("cabinet.sqlite")   # ouvrir la base
cursor = conn.cursor()

# Requête simple
resultats = cursor.execute(
    "SELECT id, nom FROM animal ORDER BY nom LIMIT 3"

for r in resultats:

# Requête avec paramètre (? remplace la valeur, évite les injections SQL)
resultats = cursor.execute(

    (2,)    # tuple même pour un seul paramètre !

Comparer des dates au format AAAAMMJJ

Les dates sont stockées sous forme de chaînes au format "AAAAMMJJ" (ex. "20241024"). Ce format a une propriété précieuse : l'ordre alphabétique = l'ordre chronologique. On peut donc comparer des dates avec >, <, >= en Python comme en SQL, sans conversion.

Exemple 
"20251125" > "20241024"   # True  → 2025 après 2024
"20240101" < "20240102"   # True  → 2 jan. après 1 jan.
# En SQL :
WHERE consultation.date > '20240923'   # toutes les dates après le 23/09/2024

Question 1 — Écrire normalisation_tel

Énoncé — Question 1

Q1 Écrire la fonction normalisation_tel(tel) qui prend un numéro de téléphone (chaîne de caractères quelconque) et renvoie uniquement ses chiffres, dans l'ordre, sous forme de chaîne. 

normalisation_tel("(0)2 12 99 90 12")   # → "0212999012"
normalisation_tel("0.6.12.99.90.12")    # → "0612999012"
normalisation_tel("06-12-99-90-12")     # → "0612999012"

Aide méthodologique

Démarche
  1. Initialiser une chaîne résultat vide : resultat = ""
  2. Parcourir chaque caractère c de tel avec une boucle for.
  3. Si c.isdigit() est vrai, ajouter c à resultat.
  4. Renvoyer resultat.

    resultat = ""

Question 2 — Jeu de tests de validation_tel

Énoncé — Question 2

Q2 La fonction validation_tel(tel) est fournie et déjà correcte. Écrire un jeu de tests (assert) qui vérifie son bon fonctionnement.

Rappel — logique de validation_tel

def validation_tel(tel):
    if len(tel) != 10:      # doit avoir exactement 10 chiffres
        return False
    if tel[0] != "0":       # doit commencer par 0
        return False
    if tel[1] != "6" and tel[1] != "7":  # 2e chiffre : 6 ou 7 uniquement
        return False
    return True

Éditeur Python — Question 2
Correction — Question 2

Stratégie : un bon jeu de tests couvre chaque branche du code. validation_tel a 3 conditions de rejet — il faut un test qui déclenche chacune d'elles. Tester uniquement les cas True sans tester les cas False n'est pas suffisant.

Point de vigilance : tester la chaîne vide "" (longueur 0 → False) permet de s'assurer qu'il n'y a pas d'erreur d'index avec tel[0] et tel[1] quand la longueur est insuffisante. Python évalue les conditions if dans l'ordre — si len != 10 est vrai, on renvoie False avant d'accéder à tel[0].

Question 3 — Écrire consultation_vaccination_chat

Énoncé — Question 3

Q3 En s'inspirant de proprietaires_animaux_nes_apres, écrire consultation_vaccination_chat(date) qui renvoie les consultations de vaccination de chats dont la date est supérieure à date.

Champs à récupérer (dans l'ordre) :

  1. id de l'animal
  2. nom de l'animal
  3. telephone du propriétaire
  4. date de la consultation

Trié par id d'animal puis date de consultation croissants.

Aide méthodologique

Démarche — construire la requête SQL par étapes
  1. SELECT : quels champs ? animal.id, animal.nom, proprietaire.telephone, consultation.date
  2. FROM : table principale → consultation (c'est là qu'est le motif)
  3. JOIN animal pour obtenir le nom et l'espèce → sur consultation.id_animal = animal.id
  4. JOIN proprietaire pour obtenir le téléphone → sur animal.id_proprietaire = proprietaire.id
  5. WHERE : deux conditions → animal.espece = 'chat' ET consultation.motif = 'vaccination' ET consultation.date > ?
  6. ORDER BY : animal.id, consultation.date

    WHERE animal.date_naissance > ?
    ORDER BY proprietaire.nom, proprietaire.prenom;

    )
    return list(resultat)

import sqlite3

DB_PATH = "cabinet.sqlite"

def consultation_vaccination_chat(date):
    """
    Renvoie les consultations de vaccination de chats dont la date > date.
    Résultat trié par id_animal puis date.
    """
    conn = sqlite3.connect(DB_PATH)
    cursor = conn.cursor()
    resultat = cursor.execute(
        """
    SELECT animal.id, animal.nom, proprietaire.telephone, consultation.date
    FROM consultation
        JOIN animal ON consultation.id_animal = animal.id
        JOIN proprietaire ON animal.id_proprietaire = proprietaire.id
    WHERE animal.espece = 'chat'
      AND consultation.motif = 'vaccination'
      AND consultation.date > ?
    ORDER BY animal.id, consultation.date;
    """,
        (date,),
    )
    return list(resultat)

Ordre des JOIN : on commence par consultation car c'est la table qui filtre sur motif et date. On joint ensuite animal pour filtrer sur espece, puis proprietaire pour récupérer le téléphone.

Le tuple (date,) : même pour un seul paramètre ?, Python exige un tuple. La virgule finale est obligatoire : (date) sans virgule n'est qu'une parenthèse, pas un tuple.

Question 4 — Analyser et corriger derniere_vaccination

Énoncé — Question 4

Q4 La fonction derniere_vaccination(consultations) est fournie mais buguée : les tests échouent. Expliquer le problème et proposer une correction.

Code fourni (bugué)

def derniere_vaccination(consultations):
    derniere = {}
    for consult in consultations:
        id_animal = consult[0]
        date = consult[3]
        if id_animal not in derniere:
            derniere[id_animal] = consult
        elif date < derniere[id_animal][3]:   # ← BUG
            derniere[id_animal] = consult
    return derniere

Analyse du bug

Identification du bug — opérateur de comparaison inversé

La condition elif date < derniere[id_animal][3] met à jour le dictionnaire quand la date courante est antérieure à celle déjà stockée. C'est l'inverse de ce qu'on veut : on cherche la dernière (la plus récente), donc il faut mettre à jour quand la date courante est postérieure.

La correction : remplacer < par >.

Trace sur Plume (id=16)
Étapeconsultdatederniere[16] actuelCondition <Action (bugué)Condition >Action (corrigé)
1ère occurrence (16, "Plume", …, "20241024") 20241024 absent — Stocker 20241024 — Stocker 20241024
2ème occurrence (16, "Plume", …, "20251125") 20251125 20241024 20251125 < 20241024 → False ❌ Ne met pas à jour 20251125 > 20241024 → True ✅ Stocke 20251125

Avec le bug, derniere[16] vaut "20241024" (la première consultation). Après correction, il vaut "20251125" (la plus récente). ✓

Éditeur Python — Question 4
Correction — Question 4

Une seule lettre à changer : < → >. Ce type de bug (opérateur inversé) est très courant et particulièrement difficile à repérer à la lecture car le code semble logique. Seul un test avec des données ayant plusieurs consultations pour le même animal révèle l'erreur.

Point de robustesse : cette fonction suppose que les consultations sont triées par date croissante (ce que garantit l'ORDER BY de la Q3). Si les données n'étaient pas triées, l'algorithme fonctionnerait quand même grâce à la comparaison explicite — c'est une force de cette approche.

Pièges fréquents & astuces de vérification

Piège 1 — Oublier le tuple pour un seul paramètre SQL

En Python, (date) entre parenthèses n'est pas un tuple — c'est juste date avec des parenthèses inutiles. Pour un tuple à un élément, la virgule finale est obligatoire : (date,). Sans elle, sqlite3 lève une ProgrammingError. 

# FAUX — n'est pas un tuple
cursor.execute("... WHERE date > ?", (date))

# JUSTE — tuple à un élément
cursor.execute("... WHERE date > ?", (date,))
Piège 2 — Oublier de convertir le curseur en liste

cursor.execute() renvoie un itérable paresseux (curseur), pas une liste. On ne peut le parcourir qu'une seule fois. Si tu le renvoies directement sans list(...), l'appelant peut se retrouver avec un objet vide dès le deuxième accès. 

# FAUX — renvoie un curseur, pas une liste
return cursor.execute("SELECT ...")

# JUSTE — convertit en liste réutilisable
return list(cursor.execute("SELECT ..."))
Piège 3 — Bug de derniere_vaccination : < vs >

Pour trouver le maximum d'une série (la date la plus récente), la condition de mise à jour est > (plus grand). Pour le minimum (date la plus ancienne), ce serait <. Confondre les deux produit un résultat plausible mais faux — c'est le bug de ce sujet. 

# Trouver le maximum (date la plus récente)
if date > derniere[id_animal][3]:
    derniere[id_animal] = consult   # on garde la plus grande
Piège 4 — Ambiguïté de noms dans le SELECT

Plusieurs tables peuvent avoir des colonnes de même nom. Par exemple, consultation.date et animal.date_naissance sont des dates, et toutes les tables ont un champ id. Sans préfixe (animal.id, consultation.date…), SQLite peut renvoyer la mauvaise colonne ou lever une AmbiguousColumnError. Toujours préfixer les colonnes dans les requêtes avec JOIN.

Piège 5 — Ne tester que les cas True dans validation_tel

Un jeu de tests qui vérifie uniquement les numéros valides (assert validation_tel("0612345678") == True) ne prouve rien sur les rejets. La fonction pourrait renvoyer True pour tout, et les tests passeraient. Il faut impérativement tester chaque condition de rejet avec un exemple qui la déclenche.

Astuces de vérification
  • Vérifier normalisation_tel avec des cas extrêmes : une chaîne sans chiffre ("abc---") doit renvoyer "", une chaîne uniquement de chiffres ("1234") doit être renvoyée telle quelle.
  • Pipeline complet : validation_tel(normalisation_tel(tel_brut)) est le vrai test d'intégration. Par exemple, normalisation_tel("06.37.97.66.64") → "0637976664" → validation_tel("0637976664") → True.
  • Tester derniere_vaccination avec un seul animal : une liste d'une seule consultation doit renvoyer un dictionnaire à une entrée — facile à vérifier à la main.
  • Compter les lignes SQL : après avoir écrit la requête Q3, vérifier que len(consultation_vaccination_chat("20240923")) == 118 est le premier test de sanité.

Récapitulatif — Ce que tu dois retenir
  • c.isdigit() renvoie True pour les chiffres 0–9, False pour tout le reste.
  • Normalisation : parcourir la chaîne, accumuler les caractères qui passent isdigit().
  • Tester validation_tel : couvrir chaque condition de rejet (len ≠ 10, tel[0] ≠ '0', tel[1] ∉ {6,7}).
  • Dates AAAAMMJJ : ordre alphabétique = ordre chronologique, comparaison directe possible.
  • Requête Q3 : deux JOIN (consultation → animal → proprietaire), trois filtres WHERE, un ORDER BY double.
  • Paramètre SQL : toujours un tuple, même à un élément → (date,) avec virgule.
  • Bug Q4 : < au lieu de > — pour garder le maximum, la condition est >.
