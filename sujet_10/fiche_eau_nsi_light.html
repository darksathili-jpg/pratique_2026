1 — Rappels : listes de dictionnaires

Imagine une feuille de tableur où chaque ligne est un relevé horodaté. En Python, on représente cela par une liste de dictionnaires : chaque dictionnaire est une ligne, chaque clé est un nom de colonne.

Rappel – accéder aux données 
# Une liste de mesures
mesures = [
    {"jour": "2025-02-04", "heure": "06:00", "chaude": 4, "froide": 6},
    {"jour": "2025-02-04", "heure": "07:00", "chaude": 6, "froide": 8},
]

# Parcourir toutes les mesures
for m in mesures:
    print(m["jour"], m["chaude"] + m["froide"])

# Filtrer les mesures d'un jour précis
for m in mesures:
    if m["jour"] == "2025-02-04":
        print(m)  # ne traiter que ce jour

Filtrer, accumuler, compter une séquence

Trois opérations reviennent constamment sur des listes de dictionnaires :

Patron 1 – Somme conditionnelle 
total = 0
for m in donnees:
    if m["jour"] == jour:
        total += m["chaude"] + m["froide"]
Patron 2 – Compteur de séquence consécutive

Pour détecter une suite d'éléments consécutifs vérifiant une condition :

compteur = 0          # nb de True consécutifs courants
for valeur in liste:
    if condition(valeur):
        compteur += 1
        if compteur >= 3:
            return True   # seuil atteint !
    else:
        compteur = 0      # réinitialisation dès qu'on voit False
return False

La clé : remettre le compteur à zéro dès qu'une valeur ne vérifie pas la condition. Sans cette réinitialisation, on compterait des occurrences non-consécutives.

Patron 3 – Renvoyer None si aucune donnée 
trouve = False
total = 0
for m in donnees:
    if m["jour"] == jour:
        total += m["chaude"] + m["froide"]
        trouve = True
if not trouve:
    return None
return total

Question 1 — Écrire total_conso

Énoncé — Question 1

Q1 Écrire la fonction total_conso(donnees, jour) qui renvoie la consommation totale d'eau (chaude + froide) pour toutes les mesures du jour donné.

Par convention : si aucune mesure n'existe pour ce jour, renvoyer None. 

>>> total_conso(donnees, "2025-02-04")
33
>>> total_conso(donnees, "2025-12-25")
>>>                           # renvoie None

Aide méthodologique

Démarche pas à pas
  1. Initialiser total = 0 et un booléen trouve = False.
  2. Parcourir donnees avec une boucle for.
  3. Pour chaque mesure, tester si m["jour"] == jour.
  4. Si oui : ajouter m["chaude"] + m["froide"] à total, passer trouve à True.
  5. Après la boucle : si trouve est False, renvoyer None, sinon renvoyer total.
Indice 1

Pour accéder au jour d'une mesure m : m["jour"].
Pour calculer la consommation totale : m["chaude"] + m["froide"].
Attention aux guillemets : les clés sont des chaînes de caractères.

Indice 2

Une erreur fréquente : renvoyer 0 quand aucune mesure n'existe. Or l'énoncé impose None. Utilise un drapeau booléen : 

trouve = False
for m in donnees:
    if m["jour"] == jour:
        ...
        trouve = True
if not trouve:
    return None
return total
Éditeur Python — Question 1
Correction — Question 1

Vérification manuelle pour le 04 : \((2+3) + (1+2) + 0 + 0 + (0+1) + 0 + (4+6) + (6+8) = 5+3+0+0+1+0+10+14 = 33\). ✓

Pourquoi un drapeau et pas juste total == 0 ? Parce qu'un jour avec des mesures toutes nulles renverrait total = 0 — ce n'est pas None. Les deux situations sont différentes : « jour connu, mais aucune consommation » ≠ « jour inconnu ».

Question 2 — Écrire fuite_possible

Énoncé — Question 2

Q2 Écrire la fonction fuite_possible(donnees, jour) qui renvoie True si au moins 3 mesures consécutives entre "00:00" et "05:00" (inclus) ont une consommation totale non nulle, False sinon.

Exemples de l'énoncé :

Heure00:0001:0002:0003:00Résultat
Conso5303→ False (pas 3 consécutifs)
Conso2110→ True (3 consécutifs)

Aide méthodologique

Démarche — Compteur de séquence
  1. Filtrer les mesures : garder uniquement celles du jour voulu ET dont l'heure est ≤ "05:00".
  2. Initialiser un compteur consec = 0.
  3. Pour chaque mesure filtrée :
    • Si consommation totale > 0 : incrémenter consec. Si consec >= 3, renvoyer True.
    • Sinon : remettre consec = 0 (réinitialisation indispensable).
  4. Après la boucle : renvoyer False.
Pourquoi filtrer sur l'heure ?

L'énoncé précise : mesures entre 00:00 et 05:00 inclus. Comparer des chaînes "HH:MM" fonctionne ici car l'ordre lexicographique coïncide avec l'ordre chronologique (toutes les heures sont sur 2 chiffres). Donc m["heure"] <= "05:00" est valide.

Indice 1 
for m in donnees:
    if m["jour"] == jour and m["heure"] <= "05:00":
        conso = m["chaude"] + m["froide"]
        # tester conso ici
Indice 2 
def fuite_possible(donnees, jour):
    consec = 0
    for m in donnees:
        if m["jour"] == jour and m["heure"] <= "05:00":
            conso = m["chaude"] + m["froide"]
            if conso > 0:
                consec += 1
                if consec >= 3:
                    return True
            else:
                consec = 0    # ← ne pas oublier !
    return False
Éditeur Python — Question 2
Correction — Question 2

Trace pour le 05/02 : les heures 00:00 à 05:00 donnent les totaux 0, 2, 2, 2, 0, 0. Compteur : 0 → 0 (réinit) → 1 → 2 → 3 ≥ 3 → True.

Pourquoi comparer les heures comme des chaînes ? Le format "HH:MM" garantit que l'ordre alphabétique = ordre chronologique. "05:00" < "06:00" est vrai en Python. C'est une propriété du format ISO 8601.

Question 3 — Analyser et corriger lissage_conso

Énoncé — Question 3

Q3 La fonction lissage_conso est censée calculer une moyenne glissante sur 3 valeurs. Testée avec [10, 20, 30, 40, 50], elle produit un résultat incorrect. Identifier le bug et le corriger.

Règles attendues :
— Élément du bord gauche (index 0) : moyenne du 1er et du 2e.
— Élément du bord droit (dernier index) : moyenne du dernier et de l'avant-dernier.
— Éléments intermédiaires : moyenne du précédent, de l'actuel et du suivant.

Qu'est-ce qu'une moyenne glissante ?

Imagine que tu veuilles « lisser » la courbe de consommation : au lieu d'afficher la valeur brute de chaque heure, tu affiches la moyenne de cette heure avec ses voisines. Les pics ponctuels sont atténués, la tendance générale devient plus lisible.

Exemple attendu sur [10, 20, 30, 40, 50]
IndexCalcul attenduRésultat attendu
0 (bord gauche)\(\frac{10 + 20}{2} = 15\)15.0
1 (intermédiaire)\(\frac{10 + 20 + 30}{3} = 20\)20.0
2 (intermédiaire)\(\frac{20 + 30 + 40}{3} = 30\)30.0
3 (intermédiaire)\(\frac{30 + 40 + 50}{3} = 40\)40.0
4 (bord droit)\(\frac{40 + 50}{2} = 45\)45.0

Résultat attendu : [15.0, 20.0, 30.0, 40.0, 45.0]

Analyse du bug dans le code fourni

Voici le code fourni. Lis-le attentivement ligne par ligne avant de regarder l'explication : 

def lissage_conso(valeurs):
    lisse = []
    for i in range(len(valeurs)):
        if i == 0:
            m = (valeurs[i] + valeurs[i+1]) / 2
        elif i == len(valeurs)-1:
            m = (valeurs[i-1] + valeurs[i]) / 2
        else:
            m = (valeurs[i-1] + valeurs[i] + valeurs[i+1]) / 2  ← BUG ICI
        lisse.append(m)
    return lisse
Identification du bug

Le cas intermédiaire calcule la somme de 3 valeurs mais divise par 2 au lieu de 3. Il additionne le précédent, l'actuel et le suivant — soit 3 termes — mais le diviseur reste 2, celui des cas de bord.

Exemple concret avec i = 2 (valeur 30) :
Code bugué : (20 + 30 + 40) / 2 = 45.0 ← faux
Attendu : (20 + 30 + 40) / 3 = 30.0 ← juste

Visualisation de l'erreur — index 2 sur [10, 20, 30, 40, 50]

← les 3 cases bleues/vertes sont additionnées

Somme correcte : 20 + 30 + 40 = 90

Indice

Il suffit de changer le diviseur du cas else de 2 à 3 :
m = (valeurs[i-1] + valeurs[i] + valeurs[i+1]) / 3

Éditeur Python — Question 3
Correction — Question 3

Une seule ligne change : / 2 → / 3 dans le bloc else. C'est un bug discret mais aux conséquences importantes : toutes les valeurs intermédiaires sont gonflées d'un facteur 1,5, ce qui rend la courbe lissée inexploitable.

Stratégie de test : le test avec [5, 5, 5, 5] (toutes les valeurs identiques) est particulièrement robuste — le lissage ne doit rien changer. Si ce test échoue, le bug est mathématiquement garanti de fausser tous les résultats.

Question 4 — Identifier et traiter le cas limite oublié

Énoncé — Question 4

Q4 La fonction lissage_conso gère déjà le cas où la liste contient exactement deux valeurs. Identifier un autre cas limite non géré et proposer une correction.

Les cas limites d'une liste

Quand une fonction reçoit une liste en paramètre, elle doit se demander : « Que se passe-t-il si la liste est vide ? Si elle a un seul élément ? » Ces situations sont les cas limites (ou cas dégénérés) — les plus susceptibles de provoquer des erreurs inattendues.

Cas limite oublié — liste vide []

Avec une liste vide, range(len([])) = range(0) : la boucle ne s'exécute pas. La fonction renvoie une liste vide [] sans erreur. Cela peut sembler correct, mais le comportement n'est pas documenté et peut surprendre l'appelant. Ce n'est pas le cas le plus critique.

Le vrai danger : liste à un seul élément [x].

Avec [42], l'index 0 déclenche le cas i == 0, qui calcule (valeurs[0] + valeurs[1]) / 2. Mais valeurs[1] n'existe pas → IndexError: list index out of range.

Trace d'exécution sur lissage_conso([42]) 
lissage_conso([42])
# i = 0
# i == 0 → m = (valeurs[0] + valeurs[1]) / 2
#                              ↑
#                   IndexError ! valeurs[1] n'existe pas
#                   len(valeurs) == 1, indices valides : {0} uniquement

Solution : traiter les cas limites en entrée de fonction

Indice 1

Pour une liste vide → renvoyer une liste vide.
Pour une liste à un seul élément → la moyenne d'un seul élément, c'est l'élément lui-même. Renvoyer [float(valeurs[0])].

Indice 2 
def lissage_conso(valeurs):
    if len(valeurs) == 0:
        return []
    if len(valeurs) == 1:
        return [float(valeurs[0])]
    # ... reste du code inchangé ...
Éditeur Python — Question 4
Correction — Question 4 (version finale complète)

Les deux gardes en tête de fonction (appelées early returns) permettent de traiter les cas dégénérés avant la boucle principale. Cela évite de complexifier la logique interne avec des conditions supplémentaires.

Bonne pratique : après avoir corrigé un bug, toujours relancer les tests existants pour vérifier qu'on n'a rien cassé — c'est ce qu'on appelle un test de non-régression. C'est pourquoi test_lissage_cas_limites inclut à la fois les nouveaux cas et le cas classique de la Q3.

Pièges fréquents & astuces de vérification

Piège 1 — Renvoyer 0 au lieu de None pour un jour inconnu

Si aucune mesure n'existe pour un jour, la somme vaut 0 sans jamais incrémenter le drapeau. Renvoyer 0 est sémantiquement faux : cela signifierait « zéro litre consommé », alors que la réalité est « aucune donnée ». Ce genre d'ambiguïté peut provoquer des bugs silencieux dans le code appelant. La convention None oblige l'appelant à tester explicitement.

Piège 2 — Oublier de réinitialiser le compteur de séquence

Dans fuite_possible, si tu oublies consec = 0 quand la consommation est nulle, le compteur continue d'accumuler des occurrences non-consécutives. Sur les données du 04/02 (5,3,0,0,1,0), le compteur atteindrait 3 (2 + réutilisation) et renverrait True à tort.

Piège 3 — Ne pas filtrer sur l'heure dans fuite_possible

Les données du 04/02 incluent des mesures à 06:00 et 07:00 avec une forte consommation. Sans le filtre m["heure"] <= "05:00", ces heures matinales seraient prises en compte et pourraient déclencher un faux positif. La fuite nocturne ne concerne que les heures de sommeil, pas le matin.

Piège 4 — Le bug du /2 dans lissage_conso est subtil

Le code fourni donne des résultats numériquement plausibles — juste trop grands — ce qui le rend difficile à repérer sans test rigoureux. Seul un test avec une valeur attendue pré-calculée révèle l'erreur. C'est pourquoi les tests de la Q3 doivent impérativement porter sur des éléments intermédiaires (ni premier, ni dernier), car les bords sont calculés correctement avec /2.

Piège 5 — Tester uniquement les cas normaux, oublier les extrêmes

lissage_conso([42]) lève une IndexError, mais si on ne teste jamais un singleton, le bug reste invisible en production. La règle d'or : toujours écrire au moins un test avec une liste vide, un singleton et deux éléments pour toute fonction recevant une liste.

Astuces de vérification
  • Vérifier total_conso à la main : somme tous les chaudes et froides du 04/02 dans les données : \(2+3+1+2+0+0+0+0+0+1+0+0+4+6+6+8 = 33\). Compte bien 16 valeurs pour 8 mesures.
  • Test symétrique pour lissage : une liste de valeurs identiques [k, k, k, k] doit donner [k.0, k.0, k.0, k.0]. Le lissage d'un signal constant doit laisser le signal inchangé.
  • Test de taille pour lissage : len(lissage_conso(v)) == len(v) doit toujours être vrai, quelle que soit la longueur de v — y compris pour [] et [x].
  • Test de l'heure limite : vérifier que 05:00 est inclus dans la détection (l'énoncé dit « entre 00:00 et 05:00 inclus »). Construire un jeu avec 2 consécutifs avant 05:00 et un troisième à 05:00.

Mini-exercices de vérification

Mini-exercice A — Tracer la fuite à la main

Pour la séquence de consommations nocturnes suivante, dis si fuite_possible renvoie True ou False :

  1. [0, 1, 2, 0, 3, 4, 5]
  2. [1, 2, 3, 0, 0, 0]
  3. [0, 0, 0, 1, 2, 3]
  1. False — max consécutifs non nuls : 2 (positions 4-5 → 3,4 puis 5 ne compte pas car c'est la suite). Trace : 0→réinit, 1→1, 2→2, 0→réinit, 3→1, 4→2, 5→3. True ! (3 consécutifs en fin)
  2. True — 1,2,3 : trois consécutifs dès le début. Compteur : 1→2→3 ≥ 3.
  3. True — 0→réinit, 0→0, 0→0, 1→1, 2→2, 3→3 ≥ 3.
Mini-exercice B — Lissage mental

Calcule à la main le résultat de lissage_conso([0, 6, 0]) (version corrigée) :

  • Index 0 (bord) : (0 + 6) / 2 = 3.0
  • Index 1 (intermédiaire) : (0 + 6 + 0) / 3 = 2.0
  • Index 2 (bord) : (6 + 0) / 2 = 3.0

Résultat : [3.0, 2.0, 3.0] — le pic à 6 est bien atténué.

Récapitulatif — Ce que tu dois retenir
  • Accéder aux données d'un dictionnaire : m["clé"] — guillemets obligatoires, casse exacte.
  • Renvoyer None (pas 0) quand aucune donnée n'existe pour le jour demandé.
  • Compteur de séquence : incrémenter si condition vraie, remettre à zéro sinon.
  • Comparer des heures "HH:MM" comme des chaînes est valide (ordre lexicographique = ordre chronologique).
  • Bug de lissage_conso : division par 2 au lieu de 3 pour les éléments intermédiaires.
  • Cas limite oublié : liste à un seul élément → IndexError car valeurs[1] n'existe pas.
  • Toujours tester : liste vide, singleton, deux éléments, et cas général.
